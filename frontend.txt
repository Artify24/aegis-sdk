# Aegis Cloud — Frontend Dashboard Prompt
> Use this prompt with any AI coding tool (Cursor, Bolt, v0, etc.) to build the full frontend.
---
## Project Overview
Build a **production-grade enterprise dashboard** called **"Aegis Cloud"** — the control plane for the Aegis AI Agent Governance SDK. This is a developer-facing SaaS platform where AI engineers register, create projects, generate API keys, and monitor every execution of their AI agents in real-time. Think **Datadog meets Vercel meets Stripe Dashboard** — dark, sharp, data-dense, and obsessively functional.
**Tech Stack:** Next.js 14+ (App Router), TypeScript, Tailwind CSS, shadcn/ui, Recharts (charts), Framer Motion (animations), Lucide icons. State management via React Context + `fetch`. No Redux.
**Backend:** FastAPI running at `http://localhost:8000`. All API calls use JSON. Auth is JWT Bearer tokens stored in `localStorage` (access + refresh tokens).
**🚨 CRITICAL DESIGN MANDATE — READ CAREFULLY 🚨**
This UI must scream "Enterprise-grade observability tool built by a top-tier design team." It CANNOT look like a generic Bootstrap template, a weekend hackathon project, or a basic shadcn default installation. You must apply obsessive attention to detail, spacing, and micro-interactions.
1. **Color Palette:** Pure dark mode only. Backgrounds should use `zinc-950` and `zinc-900`. Borders should be `zinc-800` to `white/5`. Use an aggressive, cyberpunk-adjacent accent gradient (Electric Cyan `#06B6D4` to Royal Blue `#3B82F6`) for primary buttons, active states, and glowing accents.
2. **Typography:** Use `Inter` for all UI text with extremely disciplined font weights (tight tracking for headers, high contrast for labels vs values). Use `JetBrains Mono` or `Fira Code` exclusively for execution IDs, tool names, log outputs, and API keys. 
3. **Materials & Depth:** Implement subtle glassmorphism (translucent backgrounds with `backdrop-blur-md`) for modals, dropdowns, and sticky headers. Cards should feel flat but elevate slightly on hover with a soft, diffused cyan shadow (`shadow-[0_0_15px_rgba(6,182,212,0.1)]`).
4. **Data Density:** This is an observability tool for developers. Do not use massive padding that wastes space. Keep tables and lists compact, dense, and highly scannable, similar to AWS or Datadog.
5. **Micro-animations (Framer Motion):** Every interaction needs a subtle ease. Hovering a card should scale it by `1.01`. New execution events streaming via WebSocket must slide down and fade in elegantly. Page transitions should be instant but smooth.

## Authentication System
### API Endpoints
```
POST /api/auth/register     — Body: { username, email, password } → UserResponse
POST /api/auth/login        — Body: { email, password } → { access_token, refresh_token, token_type, user: { user_id, username, email } }
POST /api/auth/refresh      — Header: Authorization: Bearer <refresh_token> → { access_token }
POST /api/auth/logout       — Returns: { message }
GET  /api/auth/me           — Header: Bearer <access_token> → UserResponse { user_id, username, email, created_at, is_active, is_verified, plan }
GET  /api/auth/verify-token — Header: Bearer <access_token> → { status: "valid", email }
```
### Pages Required
1. **`/login`** — Email + password form. Clean centered card on a dark gradient background with subtle animated grid pattern or particles. "Don't have an account? Sign up" link. Error handling with toast notifications. On success, store both `access_token` and `refresh_token` in localStorage, redirect to `/dashboard`.
2. **`/register`** — Username + email + password form. Same visual treatment as login. On success, auto-login and redirect to dashboard. The backend automatically creates a default workspace for new users.
3. **Auth Context Provider** — Wrap the entire app. Auto-refresh token on 401 responses. Expose `user`, `isAuthenticated`, `login()`, `logout()`, `register()`. Redirect unauthenticated users to `/login`. Refresh token flow: if a request fails with 401, call `POST /api/auth/refresh` with the stored refresh token, update the access token, and retry the original request.

## Dashboard Pages
### 1. Overview (`/dashboard`)
The main dashboard after login. Shows workspace-level KPIs and charts.
**API Endpoints:**
```
GET /api/workspaces                          → list of workspaces
GET /api/workspaces/{id}/summary             → { total_projects, total_executions, blocked_executions, success_rate, avg_duration_ms }
GET /api/analytics/kpis?workspace_id=X       → { activeProjects, totalExecutions, totalToolCalls, tokenUsage, totalCost, avgLatency, successRate, blockedExecutions }
GET /api/analytics/charts?workspace_id=X&days=14  → { tokens: [{date, value}], cost: [{date, value}], sessions: [{date, value}], tools: [{date, value}] }
GET /api/analytics/tool-mix?workspace_id=X   → [{ n: "tool_name", v: count }]
```
**Layout:**
- **Top row**: 4-6 KPI cards in a grid — "Total Executions", "Success Rate" (with green/red coloring), "Avg Latency", "Token Usage", "Active Projects", "Blocked Executions". Each card has an icon, the metric value large and bold, and a subtle label beneath.
- **Middle row**: Two charts side by side:
  - Left: **Area chart** — Executions over time (14-day trend from `sessions` data)
  - Right: **Donut/Pie chart** — Tool usage distribution (from `tool-mix` endpoint)
- **Bottom row**: Two more charts:
  - Left: **Bar chart** — Token usage per day (from `tokens` data)
  - Right: **Line chart** — Cost per day (from `cost` data)
Charts should use a dark theme with the electric blue/cyan accent palette. Use `Recharts` with custom dark-themed tooltips and grid lines (`zinc-800`).

### 2. Projects (`/dashboard/projects`)
Lists all projects in the selected workspace.
**API Endpoints:**
```
GET    /api/projects?workspace_id=X          → list of Project objects
POST   /api/projects                         → Body: { workspace_id, name, description, provider?, sdk_version?, environment? }
GET    /api/projects/{id}                    → Project detail
PUT    /api/projects/{id}                    → Update project
DELETE /api/projects/{id}                    → Soft delete
```
**Project Model Fields:** `project_id (_id)`, `workspace_id`, `owner_id`, `name`, `description`, `provider` (default "OpenAI"), `sdk_version` (default "1.0.0"), `environment` ("development"/"staging"/"production"), `status` ("healthy"/"degraded"/"down"), `is_active`, `created_at`
**Layout:**
- Grid of project cards (2-3 columns). Each card shows:
  - Project name (large)
  - Description (truncated)
  - Status badge (green "Healthy", yellow "Degraded", red "Down")
  - Environment badge ("Development", "Staging", "Production")
  - Provider + SDK version in small text
  - Created date
  - Click → navigates to `/dashboard/projects/{id}`
- "Create Project" button opens a modal/drawer with form fields: name, description, provider (dropdown), environment (dropdown)

### 4. API Keys (`/dashboard/projects/{id}/keys`)
Manage API keys for a project. This is where developers get their `AEGIS_PROJECT_KEY`.
**API Endpoints:**
```
GET    /api/projects/{project_id}/keys                    → list of APIKeyResponse
POST   /api/projects/{project_id}/keys                    → Body: { name, expires_at? } → APIKeyFullResponse (includes raw_key — SHOW ONLY ONCE)
PUT    /api/projects/{project_id}/keys/{key_id}/disable   → Disable key
PUT    /api/projects/{project_id}/keys/{key_id}/enable    → Enable key
DELETE /api/projects/{project_id}/keys/{key_id}           → Delete key
POST   /api/projects/{project_id}/keys/{key_id}/rotate    → Rotate key → new APIKeyFullResponse
```
**APIKeyResponse Fields:** `key_id`, `project_id`, `workspace_id`, `name`, `key_prefix` (first 10 chars like `ag_h5wWFfwe`), `is_active`, `created_at`, `expires_at`, `last_used_at`, `created_by`
**Layout:**
- Table with columns: Name, Key Prefix (monospace, with copy icon), Status (Active/Disabled badge), Created, Last Used, Expires, Actions (Enable/Disable toggle, Rotate, Delete)
- "Create API Key" button opens modal. On creation, show the **full raw key** in a special one-time-view modal with a copy button and a warning: "This key will not be shown again. Copy it now."
- Rotate shows a confirmation dialog, then displays the new key in the same one-time-view modal
- Delete shows a destructive confirmation dialog

### 5. Executions (`/dashboard/executions`)
The core observability page. Shows every execution report streamed from the SDK.
**API Endpoints:**
```
GET /api/executions?project_id=X&filter_status=SUCCESS&risk_level=HIGH&page=1&limit=20
    → { data: [ExecutionSummary], meta: { total, page, limit, has_next } }
GET /api/executions/{execution_id}              → Full ExecutionReport
GET /api/executions/{execution_id}/timeline     → Timeline events array
GET /api/executions/{execution_id}/tool_calls   → Tool call records array
GET /api/executions/{execution_id}/governance   → Governance report object
GET /api/executions/{execution_id}/metrics      → Metrics object
```
**Execution Summary (list item) Fields:**
`execution_id`, `status` (SUCCESS/FAILED/BLOCKED/CANCELLED), `risk_level` (LOW/MEDIUM/HIGH/CRITICAL), `governance` (ALLOW/DENY/APPROVAL_REQUIRED), `duration_ms`, `tools_used` (count), `created_at`
**List Page Layout:**
- Filter bar at top: Status dropdown, Risk Level dropdown, Project selector (if not already scoped)
- Table/list with columns: Status (colored badge), Execution ID (monospace, truncated with copy), Risk Level (colored badge), Governance Decision, Duration, Tools Used, Created At
- Pagination controls at bottom
- Click a row → navigates to execution detail page
**Execution Detail Page (`/dashboard/executions/{id}`):**
This is the crown jewel of the dashboard. It should feel like a detailed forensic view of a single AI agent run. Use a multi-section layout with collapsible panels:

### 6. Policies (`/dashboard/policies`)
**API Endpoints:**
```
GET    /api/policies                          → list of Policy
POST   /api/policies                         → Body: { name, scope, description, enabled? }
PUT    /api/policies/{id}                    → Update policy
DELETE /api/policies/{id}                    → Delete policy
```
**Policy Fields:** `id`, `name`, `enabled` (boolean), `scope` ("global"/"project"/"tool"), `description`, `triggers` (count)
**Layout:**
- Table with columns: Name, Scope badge, Status (Enabled/Disabled toggle), Triggers count, Actions (Edit, Delete)
- "Create Policy" button opens modal with form
- Inline toggle for enable/disable
---
### 7. Risk Feed (`/dashboard/risk`)
Shows high-risk and blocked executions.
**API Endpoint:**
```
GET /api/risk?workspace_id=X&project_id=Y&severity=HIGH&limit=20
    → { data: [ExecutionSummary] }
```
**Layout:**
- Feed/timeline style list of risk events
- Each item shows: status badge, risk level badge, execution ID (clickable → detail page), duration, tools used, created time
- Severity filter at top (LOW, MEDIUM, HIGH, CRITICAL)
---
### 8. Logs (`/dashboard/logs`)
**API Endpoints:**
```
GET  /api/logs?level=ERROR&source=Layer1&limit=100     → list of LogEntry
POST /api/logs                                          → Create log entry
```
**LogEntry Fields:** `id`, `ts` (timestamp), `level` ("INFO"/"WARNING"/"ERROR"/"DEBUG"), `source`, `message`, `meta` (JSON object)
**Layout:**
- Filterable log viewer with level filter (INFO/WARN/ERROR/DEBUG tabs or dropdown)
- Each log line: timestamp (monospace), level badge (colored), source, message
- Expandable to show `meta` JSON
---
### 9. Real-Time WebSocket Integration
**WebSocket Endpoint:**
```
ws://localhost:8000/api/ws/project/{project_id}
```
The backend broadcasts `{ type: "new_execution", execution: ExecutionSummary }` when a new execution report is ingested.
**Implementation:**
- When viewing a project's executions page, connect to the WebSocket
- On `new_execution` message, prepend the new execution to the list with a subtle slide-in animation
- Show a toast notification: "New execution received — [status]"
- Optional: Live activity indicator (green dot) in the sidebar when connected

## Non-Functional Requirements
1. **Responsive**: Must work on 1920px+ desktops, tablets (sidebar collapses), and mobile (sidebar becomes bottom nav or hamburger)
2. **Loading States**: Use skeleton screens (shimmer effect), not spinners, for initial page loads
3. **Error Handling**: All API errors show toast notifications. Network errors show a retry banner.
4. **Empty States**: Every list page has a designed empty state with an illustration/icon and a CTA (e.g., "No executions yet. Generate an API key and start sending telemetry.")
5. **Accessibility**: Proper ARIA labels, keyboard navigation, focus indicators
6. **Dark mode only**: The entire app is dark theme. No light mode toggle needed.
7. **Animations**: Subtle Framer Motion transitions on page changes, card hovers (slight scale + shadow lift), list item entrances (stagger fade-in)
